# Neovim 快捷键说明

当前 `Leader` 键为 `,`。

这份文档记录当前配置中已经落地的快捷键设计、分组原则、常用键位，以及从旧键位迁移到新键位的对应关系。

---

## 一、设计原则

这套键位是按“手感优先”重新整理的，核心原则如下：

1. 高频操作尽量短
   但也优先服从你的肌肉记忆。当前 LSP 相关功能统一收进 `,l*`。

2. 同类功能尽量归到同一前缀
   例如：
   - `,g*` 统一表示 Git
   - `,l*` 统一表示 LSP
   - `,f*` 统一表示查找 / 搜索 / 文件
   - `,a*` 统一表示 AI 能力
   - `,t*` 统一表示终端

3. 尽量不和已有生态习惯硬碰硬
   比如：
   - 自定义功能尽量统一收进 `<leader>` 前缀，减少跨前缀记忆负担
   - `,c*` 保留给 cscope，不去和它抢前缀

4. 优先保证长期可维护
   比起“某一个键眼前最短”，更重要的是一段时间后你还能靠直觉记住整套布局。

---

## 二、整体分组

当前推荐按下面这个脑图来记：

- `,f*`：查找、搜索、文件
- `,e*`：文件树 / Explorer
- `,g*`：Git
- `,l*`：LSP 功能
- `,c*`：cscope
- `,t*`：终端
- `,a*`：AI / CodeCompanion
- `,n*`：注释生成这类少量独立动作

---

## 三、核心基础键位

### 1. 基础编辑

- `,N`
  在当前文件所在目录中新建文件。
  这个命令会自动把当前 buffer 所在目录补到命令行里，适合快速在同目录下建新文件。

- `jk`（插入模式）
  退出插入模式，回到普通模式。

- `Esc`（普通模式）
  清除搜索高亮。

### 2. 行间移动

- `j`
- `k`

这两个键现在做了“按场景切换”的处理：

- 如果你没有输入计数，比如直接按 `j` / `k`
  那么它们按“屏幕行”移动，也就是相当于 `gj` / `gk`
  这对长行折行显示时更顺手。

- 如果你输入了计数，比如 `5j` / `3k`
  那么仍然保留原生 `j` / `k` 的行为，按“真实行”移动。

这样比以前直接硬映射成 `gj/gk` 更稳，不会破坏带计数的操作习惯。

---

## 四、查找 / 搜索 / 文件：`,f*`

这一组统一处理“找东西”。

- `,b`
  打开 buffer 列表。

- `,ff`
  查找文件。

- `,fl`
  在当前 buffer 中搜索行。

- `,fw`
  打开 Treesitter 符号列表。

- `,fs`
  全局实时搜索（live grep）。

- `,fr`
  搜索光标下单词。

- `,fj`
  恢复上一次 FzfLua picker。

- `,fq`
  打开最近文件列表。

- `,fm`
  格式化当前 buffer 或当前可视选区。

为什么放在 `,f*`：

- `f` 很自然就能联想到 `find`
- 这一组都是“查找/筛选/定位”行为，放一起便于肌肉记忆

---

## 五、文件树 / Explorer

- `,e`
  打开或关闭文件树。

- `,E`
  聚焦到文件树窗口。

- `,ge`
  打开文件树中的 Git 状态视图。
  这个键虽然功能上属于 Explorer，但前缀放在 `,g*` 下，方便从 Git 语境直接进入 Git 状态视图。

为什么不用原来的 `,fe` / `,fE`：

- 文件树本质上不是“查找”
- 它更像一个独立的“资源浏览器”
- 单独给 `,e*` 后，语义更清楚，手感也更短

---

## 六、Git：`,g*`

这一组是本次整理里的重点。现在 Git 相关功能尽量都归到 `,g*`。

### 1. Git 主入口

- `,gg`
  打开 fugitive 的 Git 界面。

- `,gb`
  打开 `Git blame`。

- `,Gc`
  查看 Git 提交记录（FzfLua）。

- `,Gs`
  查看 Git 状态（FzfLua）。

### 2. Hunk 跳转

- `,gj`
  跳到下一个 hunk。

- `,gk`
  跳到上一个 hunk。

### 3. Hunk 操作

- `,gs`
  stage 当前 hunk。
  在可视模式下表示 stage 当前选中的 hunk 区间。

- `,gr`
  reset 当前 hunk。
  在可视模式下表示 reset 当前选中的 hunk 区间。

- `,gS`
  stage 整个 buffer。

- `,gu`
  undo 上一次 stage hunk。

- `,gR`
  reset 整个 buffer。

- `,gp`
  预览当前 hunk。

### 4. blame / diff / deleted

- `,gl`
  查看当前行的完整 blame 信息。

- `,gB`
  切换当前行内联 blame 显示。

- `,g=`
  对比当前内容与 index。

- `,g-`
  对比当前内容与上一个提交。

- `,gX`
  切换 deleted hunks 的显示。

### 5. 文本对象

- `ih`
  选中当前 hunk。

### 6. 从旧键位迁移

旧版里有一批 `,h*` 键位，现在已经统一迁到 `,g*`：

- `,hs` -> `,gs`
- `,hr` -> `,gr`
- `,hS` -> `,gS`
- `,hu` -> `,gu`
- `,hR` -> `,gR`
- `,hp` -> `,gp`
- `,hb` -> `,gl`
- `,hd` -> `,g=`
- `,hD` -> `,g-`

这么改的原因很直接：

- 它们本来就都是 Git 动作
- 放在 `,h*` 下记忆不够统一
- 放回 `,g*` 后，Git 功能只需要记一组

## 七、LSP

LSP 统一放到 `,l*` 下，减少和默认裸键、Git、cscope 的前缀冲突，也更符合你“自定义快捷键尽量都用 leader 开头”的习惯。

### 1. 导航与跳转

- `,ld`
  跳到定义。

- `,lr`
  查看引用。

- `,lD`
  跳到声明。

- `,la`
  Code action。

- `,ln`
  重命名符号。

- `,lm`
  查看 implementations。

- `,lt`
  查看 type definitions。

- `,ll`
  查看当前文件的 symbols。

- `,ls`
  工作区符号搜索。

- `,lx`
  当前文档诊断列表。

- `,lh`
  C/C++ 头文件 / 源文件切换。

- `,li`
  查看 incoming calls。

- `,lo`
  查看 outgoing calls。

- `,lp`
  弹出 calltree 视图。

### 2. 为什么不把 LSP 放到 `,c*`

因为你当前已经启用了 `cscope_maps.nvim`，而且它默认整套都占用 `,c*`。

所以这里刻意避开 `,c*`，避免：

- 和记忆冲突
- 和 cscope 默认键位打架
- 未来维护时越来越乱

Neovim 自带的裸键 LSP 映射保持不动；当前配置建议优先使用 `,l*` 这套自定义映射。

插入模式下还提供：

- `,lS`
  查看当前函数的参数签名提示（signature help）。

## 八、Cscope：`,c*`

这一组保留给 `cscope_maps.nvim`。

你当前除了手工加的两个键：

- `,cj`
  Cscope StackView down

- `,ck`
  Cscope StackView up

还会继承插件默认的一整套 `,c*` 快捷键。

常见包括：

- `,cs`
  查找引用

- `,cg`
  查找全局定义

- `,cc`
  查找调用者

- `,ct`
  查找文本出现

- `,ce`
  egrep 搜索

- `,cf`
  打开光标下文件名

- `,ci`
  查找 include 当前文件的文件

- `,cd`
  查找当前函数调用了哪些函数

- `,ca`
  查找赋值位置

- `,cb`
  构建 cscope 数据库

这组建议你不要再塞别的功能进来。

## 九、Treesitter Textobjects

### 1. 选择：`af` / `if` / `ac` / `ic` / `as`

- `af`
  选中整个函数。

- `if`
  选中函数内部。

- `ac`
  选中整个类。

- `ic`
  选中类内部。

- `as`
  选中当前作用域。

这几个只在：

- 可视模式 `x`
- operator-pending 模式 `o`

下使用。

例如：

- `vaf`
- `daf`
- `yif`

### 2. 跳转：`,j*` / `,k*`

#### 向后跳

- `,jf`
  下一个函数开头

- `,jc`
  下一个类开头

- `,ja`
  下一个参数

#### 向前跳

- `,kf`
  上一个函数开头

- `,kc`
  上一个类开头

- `,ka`
  上一个参数

为什么不用原来的 `[ ]`：

- `[f` / `]f` 更像 Vim 传统手法，但手感一般
- `,j*` / `,k*` 更接近“向下找 / 向上找”的身体感觉
- 对你这种明确想追求手感的使用习惯更合适

### 3. 参数交换

- `,xj`
  与下一个参数交换。

- `,xk`
  与上一个参数交换。

## 十、终端：`,t*`

现在终端统一放到 `,t*`。

- `,tn`
  新建终端。

- `,tt`
  切换终端显示 / 隐藏。

- `,tz`
  放大 / 还原当前终端或当前 buffer 的 tab 显示。

### 终端模式下

- `,te`
  从终端模式退回普通模式。

- `,th`
  从终端模式退出并隐藏终端。

为什么从原来的终端键位改走：

- 终端不应该混在 `,f*` 里
- `,m` 这种过于模糊，长期会忘
- `,t*` 非常稳定，看到就知道是 terminal

---

## 十一、AI：`,a*`

AI 相关能力统一放到 `,a*`。

### 1. 主入口

- `,ai`
  打开 AI 聊天。

- `,aa`
  打开 AI Actions。

- `,ae`
  触发 AI inline edit。

- `,ah`
  打开 AI 历史。

- `,am`
  生成 commit message。

- `,at`
  切换 AI 聊天窗口。

### 2. 可视模式下的 AI 操作

这些键在可视模式下对选中代码生效：

- `,ae`
  解释代码

- `,ao`
  优化代码

- `,ac`
  添加注释

- `,af`
  修复 Bug

- `,at`
  生成测试

说明：

这里 `,ae` / `,at` 在普通模式和可视模式下语义不同，但都仍然属于 AI 范畴：

- 普通模式更偏“入口”
- 可视模式更偏“对选区做操作”

这种复用是有意为之，目的是减少记忆负担。

## 十二、其他

- `,ng`
  用 Neogen 生成注释 / 注解模板。

- `,w`
  Leap 单窗口快速跳转。

- `,W`
  Leap 跨窗口快速跳转。

- `,?`
  打开 which-key，查看当前 buffer 的本地快捷键。

这里仍然保留少量非 `<leader>` 键位作为例外：

- `jk`：插入模式退出
- `j` / `k`：屏幕行移动
- `K`：LSP hover
- `af` / `if` / `ac` / `ic` / `as`：Treesitter textobjects
- `ih`：Git hunk textobject
- `gc` / `gcc` / `gC`：Comment.nvim 默认注释键

这些键要么是 Vim 传统用法，要么是插件生态里的通用约定，继续保留会比强行 leader 化更自然。

## 十三、关于“之前那些 `,h` 怎么办”

你之前专门问过这件事，这里单独说明。

结论是：

原来的 `,h*` 大部分已经不再作为主键位使用，而是被归并到了 `,g*`。

原因：

1. 它们本质上都是 Git hunk 相关动作
2. 放在 `,h*` 会让 Git 功能分成两堆
3. 放到 `,g*` 后，Git 的认知模型更统一

也就是说，现在你应该把：

- `,h*`

这一组理解成“旧记忆”，把新的主记忆切到：

- `,g*`

如果你想保留一段过渡期，也可以后面再给 `,h*` 加兼容别名，但当前配置里还没有加这一层兼容。

## 十四、建议的记忆方式

不要试图死记全部键位，推荐按“分组 + 高频”记：

### 第一步：先记高频

- `,w`
- `,ff`
- `,fs`
- `,e`
- `,ld`
- `,lr`
- `,la`
- `,ln`
- `,gg`
- `,gj`
- `,gk`
- `,gs`
- `,gr`
- `,tt`

### 第二步：再记分组

- `,f` 就是找东西
- `,g` 就是 Git
- `,l` 就是 LSP
- `,c` 就是 cscope
- `,t` 就是 terminal
- `,a` 就是 AI

### 第三步：最后记低频长尾

比如：

- `,lp`
- `,ge`
- `,g=`
- `,g-`
- `,ng`

这样记忆成本会低很多。
